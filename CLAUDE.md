# PeachPDF

Pure .NET HTML → PDF rendering library. No external process dependency (no Puppeteer/wkhtmltopdf/gs) — HTML parsing, CSS cascade, layout, and PDF writing all run in-process. Targets .NET 8 and .NET 10.

## Project layout

- `src/PeachPDF/` — the library. Notable subtrees: `CSS/` (CSS-OM: tokenizer, parser, value converters), `Html/Core/` (DOM, cascade, layout, paint handlers), `Html/Adapters/` (`RGraphics`/`RAdapter`/etc. abstraction layer), `Svg/` (native SVG tree builder + renderer), `PdfSharpCore/` (embedded fork of PDFsharp — see below, has its own [LICENSE](src/PeachPDF/PdfSharpCore/LICENSE.md)).
- `src/PeachPDF.Tests/` — xUnit test suite, multi-targets net8.0/net10.0.
- `src/PeachPDF.TestHarness/` — a runnable showcase app for visually exercising features (`Program.cs`).
- `src/PeachPDF.Benchmarks/` — BenchmarkDotNet project.
- Working directory for `dotnet build`/`dotnet test` is `src/` — if a shell session resets to repo root, commands silently fail with "Project file does not exist"; always confirm you're in `src/` first.
- If the .NET SDK is missing (e.g. a fresh sandboxed/web session), install it with `apt` — `sudo apt-get install -y dotnet-sdk-10.0` (add `aspnetcore-runtime-8.0` so the net8.0 target framework can run). The `dot.net/v1/dotnet-install.sh` route is blocked in this environment (the Microsoft CDN host returns 403 through the egress proxy), so apt is the working path. `global.json` pins SDK 10.0.100 (rollForward `latestFeature`), which the apt `dotnet-sdk-10.0` (10.0.1xx) satisfies.

## Critical: `dotnet test` invocation

**Always pass `--framework net8.0`.** The test project multi-targets net8.0 and net10.0; a bare `dotnet test` builds and runs the ~3000+ test suite twice in one invocation. Repeated bare invocations across a long session have crashed the user's Windows machine from cumulative build/test load. Single-target coverage is sufficient for regressions; only run net10.0 explicitly if a net10.0-specific issue is actually suspected.

```
dotnet test PeachPDF.Tests/PeachPDF.Tests.csproj --framework net8.0
dotnet test PeachPDF.Tests/PeachPDF.Tests.csproj --framework net8.0 --filter "FullyQualifiedName~Svg"
```

Batch verification — run the full suite once after a meaningful chunk of related changes, not after every individual file edit.

### Coverage

CI enforces 90% diff coverage on PRs (`diff-cover` against the PR base, via `.github/workflows/test.yml` + `src/PeachPDF.Tests/coverlet.runsettings`, Cobertura format). To reproduce locally:

```
dotnet test --collect:"XPlat Code Coverage" --settings PeachPDF.Tests/coverlet.runsettings --results-directory coverage
```

Before considering any non-trivial code change complete, run the command above and check diff coverage on the lines you changed. If new/changed code falls short of the 90% diff-coverage threshold CI enforces, add tests to close the gap before finishing — don't leave it for CI to catch.

Avoid writing tests against `PdfSharpCore.Fonts.FontFactory` (and OpenType neighbors) without care — it caches resolved fonts in `static readonly Dictionary` fields shared process-wide, and xUnit's parallel test-class execution makes new tests here a real order-dependent-flakiness risk against the rest of the suite.

## Post-change review pass

After a non-trivial code change is otherwise done (implementation + tests passing + coverage checked), spin up a review agent against the diff (uncommitted changes, or the changes in the current branch/PR) covering:

1. **C# coding conventions** — [Microsoft's C# coding conventions](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/coding-style/coding-conventions).
2. **Current C# language version** — this repo doesn't pin `LangVersion` (see `src/Directory.Build.props`), so each TFM already floats to its own SDK-default latest (net8.0 → C# 12, net10.0 → C# 14, etc.) — flag code that reads like an older-version holdover (e.g. missing collection expressions, primary constructors, pattern-matching improvements) where a newer construct would be a genuine clarity/correctness win, not a style-only rewrite.
3. **Current .NET APIs for the target frameworks** — prefer the modern API over an older equivalent still in the codebase out of habit. Since this repo multi-targets `net8.0;net10.0`, if a meaningfully-better API only exists on the newer TFM, gate it with `#if NET10_0_OR_GREATER` (falling back to the net8.0-compatible approach in `#else`) rather than skipping it — don't downgrade the whole codebase to the lowest common denominator when the newer target can do better.
4. **Current HTML/CSS/SVG spec compliance** — check changes touching parsing/cascade/layout/paint against the actual current WHATWG/W3C spec text for the relevant module(s) (HTML parsing/living standard, the specific CSS module(s) involved — Box, Text, Position, Paged Media, Fragmentation, etc. — and/or the relevant SVG module), not against assumptions or this repo's existing (possibly incomplete) behavior. Cross-check against [docs/html-css-support.md](docs/html-css-support.md) / [docs/supported-svg-features.md](docs/supported-svg-features.md) for known, already-accepted gaps before flagging something as a new defect.

Report findings the same way `/code-review` does; fix what's actually wrong, and record any newly-confirmed accepted gap in the relevant doc rather than leaving it as a silent finding.

**When a change deliberately leaves a CSS/SVG/HTML spec violation out of scope, file a GitHub issue for it** — one that states what the gap is, the specific spec rule it violates, and why it was out of scope — and reference that issue number in the accepted-gap note. An accepted gap that is a genuine spec deviation must be *tracked as an issue*, not only recorded as prose (mirroring the existing `#151`/`#152`/`#166` entries below). This complements the docs rule above: the CLAUDE.md accepted-gap note carries the `#NNN` reference, while the reader-facing `docs/**` note describes the same limitation without citing the issue.

## Testing conventions

- **A passing test that only checks PDF content-stream substrings (`/SMask`, `Tj`, `/ShadingType`, etc.) is not proof a feature renders correctly.** A token can be 100% present while the actual composed/positioned result is visually broken or blank — this exact gap let a fully-broken `<mask>` implementation pass 16/16 tests. For anything touching PDF graphics state — soft masks, patterns, clip paths, gradients, transparency groups, transforms — prefer structural/adjacency assertions (e.g. regex-checking that a `gs` and the `Do` it modifies appear on the same `cm` line) or, better, actually rasterize the output and look at it.
- **Rasterize with two renderers, not one**, when verifying transparency/soft-mask/blend-mode output — MuPDF is unusually lenient about transparency-group conformance and can render content "correctly" that a stricter, more representative engine (PDFium — Chrome/Edge's engine) refuses. Agreement between both is real evidence; a single MuPDF render that looks right is not.

  ```bash
  python3 -m pip install --quiet pymupdf pypdfium2 pillow   # once per machine

  python3 -c "
  import fitz
  doc = fitz.open('path/to/file.pdf')
  doc[PAGE_INDEX].get_pixmap(dpi=150).save('out_mupdf.png')
  "
  python3 -c "
  import pypdfium2 as pdfium
  pdf = pdfium.PdfDocument('path/to/file.pdf')
  pdf[PAGE_INDEX].render(scale=2.0).to_pil().save('out_pdfium.png')
  "
  ```
  Then view both PNGs with the Read tool and compare.
- When implementing a new SVG (or CSS) **paint** feature, a parser-level "did it parse into the right enum/value" test is not sufficient on its own — always add an integration test that would fail if the feature were a complete no-op (a prior gradient `spreadMethod` bug shipped with only a parser test and was a no-op at render time for months).
- **Layout engine changes** (`CssLayoutEngine`/`CssLayoutEngineFlex`/`CssLayoutEngineTable`/`CssLayoutEngineColumns`, `CssBox.PerformLayoutImp`) need unit tests that assert the relevant `CssBox`'s properties after layout — `Location`, `ActualRight`/`ActualBottom` (size), etc. — not just that layout completes without throwing. Assert on every box the change affects, including children when the change affects child placement (e.g. a multi-column child's `Location.X`/`Location.Y`, a flex item's sizing). Use the lightweight harness pattern in `FlexboxIntegrationTests.cs`/`MulticolLayoutIntegrationTests.cs`: build a `HtmlContainerInt` + `PdfSharpAdapter`, call `PerformLayout` directly, then walk the box tree by id/class and assert positions/sizes — no full PDF generation needed.
- **Painting changes** need unit tests that confirm the actual sequence of calls made to the `RGraphics` adapter layer — which calls, in what order — not just that painting completes or that some token shows up in the final PDF (see the content-stream-substring pitfall above). Use a test-only `RGraphics` mock (see `SpyGraphics` in `TransformIntegrationTests.cs`, `RecordingGraphics` in `CssLayoutEngineTablePageBreakTests.cs`) that overrides the relevant methods to record each invocation, then paint through it and assert on the recording. When order *across different call types* matters (e.g. background drawn before border, a clip pushed before the content it clips), record into a single ordered log rather than the separate per-call-type counts/lists those two existing mocks use — extend the pattern to fit, don't duplicate a parallel one.
- Don't copy test files verbatim from upstream `empira/PDFsharp` into `src/PeachPDF.Tests/PdfSharpCore` — this fork is a ~2016-era snapshot (write-only, no `PdfReader`/`Lexer`/`ContentReader`, no Attachments/Signatures/Security/PdfA/Forms/Metadata, different font subsystem). A 96-file verbatim batch yielded only 5 portable files; the rest had to be deleted. Fresh, fork-native tests are the better path to closing coverage gaps.

## Value normalization to know before asserting in tests

- Named CSS colors are normalized to `rgb(r, g, b)` form at parse time (`color: blue` → `CssBox.Color == "rgb(0, 0, 255)"`), except `initial` which resolves to the literal string `"black"`.
- `em` font-sizes are eagerly converted to points at cascade time in the `CssBoxProperties.FontSize` setter (relative to the parent's actual font size in pt), not kept symbolic — assert against the converted `pt` value, not the original `em` string.
- The internal layout unit is 1 PDF point, and CSS `px` resolves spec-correctly at **1px = 1/96in = 0.75pt** everywhere (`Length.PointsPerPx`, the only place the ratio lives — issue #150 removed the old "1px = 1pt except fonts" dual convention). A `width: 96px` box lays out 72 units wide; image/SVG intrinsic pixels are ×0.75 too. When writing layout tests where the unit isn't the point of the test, prefer `pt` in fixtures so expected values read literally.

## Architecture conventions

- **Don't write two independent parsers for the same CSS value grammar across layers.** If both the CSS-OM/parsing layer and a later render/resolution layer need to understand a value's grammar (tokenization + classification), extract it into one shared internal class both call — only the final numeric resolution that genuinely depends on runtime-only information (percentage-against-box-size, `calc()` evaluation) should differ between layers. Precedent: `CalcParser` (shared across `PeachPDF.CSS` ⇄ `PeachPDF.Html.Core`), `BackgroundPositionGrammar`/`BackgroundSizeGrammar` (shared between the CSS-OM converters and `BackgroundLayerResolver`).
- The `Html/Adapters` layer (`RGraphics`/`RAdapter`/`RPen`/etc.) is the abstraction boundary between layout/paint logic and the concrete PDF backend (`PdfSharpCore`). New rendering primitives (tiling, soft masks, dash patterns, etc.) get added here first, then implemented in `GraphicsAdapter`/`XGraphics`/`XGraphicsPdfRenderer`. Check whether an abstract `RGraphics` member you're adding needs updates to the test-only mock implementations (`SpyGraphics` in `TransformIntegrationTests.cs`, `RecordingGraphics` in `CssLayoutEngineTablePageBreakTests.cs`).
- Before building new PDF-writing infrastructure (patterns, soft masks, shadings), check whether `PdfSharpCore` already has an unused primitive for it (`XForm`/`PdfFormXObject`, `PdfTilingPattern`, `PdfSoftMask` have all been found pre-existing-but-uncalled at various points) rather than assuming it needs to be built from scratch.
- A `/Luminosity` soft mask's `/G` form AND the content form it masks both need their own `/Group << /S /Transparency /CS ... /I true >>` transparency-group dictionary for spec-conformant readers to actually apply the mask — `XForm`/`PdfFormXObject` don't set this automatically.

## Out of scope / accepted gaps (don't relitigate without new information)

- **Per-page horizontal reflow (issue #143) is scoped to auto-width main-column blocks.** The width seam (`CssLayoutEngine.GetBoxWidth` → `HtmlContainerInt.PageContentRightOf`, gated by `IsMainColumnContainer` = root/`<html>`/`<body>`) reflows only auto-width block content whose containing block is the main column. Each remaining case is a genuine CSS Paged Media 3 §5 ("the edges of the page area act as a containing block for the layout that occurs between page breaks") deviation, tracked as its own issue: flex ([#196](https://github.com/jhaygood86/PeachPDF/issues/196)), tables ([#197](https://github.com/jhaygood86/PeachPDF/issues/197)), multicol ([#198](https://github.com/jhaygood86/PeachPDF/issues/198)) — the horizontal analogue of the #166 engine-independence family; percentage/fixed-`width` blocks ([#199](https://github.com/jhaygood86/PeachPDF/issues/199)); content nested below the main column, incl. a descendant that begins on a different page than its multi-page auto-width ancestor ([#200](https://github.com/jhaygood86/PeachPDF/issues/200)); the initial containing block resolving against the base page area rather than the first page's `:first`-overridden area ([#201](https://github.com/jhaygood86/PeachPDF/issues/201)); and named-page (`page: <name>`) L/R overrides, whose width→height→page-name feedback the bounded reflow loop doesn't drive to convergence ([#202](https://github.com/jhaygood86/PeachPDF/issues/202)). NOT a gap (spec-correct, don't file): a box spanning a page boundary keeps its start-page measure across fragments — CSS Fragmentation 3 shares one inline size across a box's fragments.
- SMIL animation, scripting, `<cursor>`/`<view>`, legacy SVG glyph-outline fonts, `<filter>`+`fe*` primitives, `<foreignObject>`, `icc-color` in SVG.
- `letter-spacing`'s CSS Text Level 3 §7.2 "not at the start/end of a *line*" exemption is not implemented — a word's last-on-line status isn't known until `FlowBox`'s wrapping decision completes, after which both layout (`CssBox.MeasureWordsSize`/`ApplyFirstLineStyleOverride`/`RemeasureWordsTail`) and paint (the PDF `Tc` operator, which applies unconditionally to every glyph shown) would need to retroactively drop the line-boundary gap. Since layout and paint now consistently agree on including a word's trailing letter-spacing (fixed — see `LetterWordSpacingLayoutIntegrationTests.cs`), this is a pre-existing, sub-pixel, invisible-in-practice deviation (a genuinely-last-on-line word reserves/paints one extra letter-spacing unit of blank trailing space no browser would show), not a newly-introduced one.
- `:visited`/`:active` (and other interaction-state pseudo-classes) never match, by design — there is no browsing history or interaction/hover state in a static PDF renderer. `CssData.DoesSelectorMatch(PseudoClassSelector, ...)` intentionally only matches `:root` and `:link`.
- Named-page (`page: <name>`) activation and **reversion** are honored only for normal block-flow content. The used value of `page` is now tree-based (CSS Paged Media Level 3 §3) and reverts for content leaving a named subtree — fixed for the block-flow forced-break path in `CssBox.PerformLayoutImp` (issue #126), but *not* for children positioned independently by `CssLayoutEngineFlex`/`CssLayoutEngineTable`/`CssLayoutEngineColumns`, which don't route through that path (same engine-independence family as the CSS Fragmentation §5.2 margin-truncation gap above). A `page` change/reversion among flex items, table cells, or multicol children may not begin/revert on a fresh page. Filed as [issue #166](https://github.com/jhaygood86/PeachPDF/issues/166). See `NamedPageLayoutIntegrationTests`/`NamedPageGeometryAttributionTests` for the fixed block-flow behavior (used-value reversion, band restoration, margin-box suppression no longer leaking).
- SVG text cross-ancestor `font-*` inheritance: `font-family`/`font-size`/`font-weight`/`font-style` inherit only *within* a `<text>` subtree (from `<text>` down through its `<tspan>`s), not from an ancestor `<g>`/`<svg>` as real SVG/CSS inheritance requires — `SvgTreeBuilder`'s `FontContext` is threaded only through text-run ancestors. (The SVG `<style>`/cascade unification below deliberately did *not* make `fill`/`stroke`/`font-*` cascaded `CssBox` properties, so this stayed as-is; `var()` custom properties *do* now inherit cross-ancestor.) Filed as [issue #188](https://github.com/jhaygood86/PeachPDF/issues/188) (gap 2).
- em/rem-relative `calc()` inside an SVG length uses a fixed 16px approximation for `em`/`rem` (the same approximation `SvgValueParsers.ParseLength` already uses for a plain `em`/`rem` SVG length), since arbitrary SVG geometry attributes have no live font-size context — only where the element's real font context is threaded would it differ. Pure-length and percentage `calc()` are exact. Filed as [issue #207](https://github.com/jhaygood86/PeachPDF/issues/207).
- `::marker` marker-box width/height/margin/padding/alignment (the CSS Lists Level 3 "marker box" layout model) is not implemented — the spec itself (§3.1.1) declares this layout "not fully defined" and restricts applicable properties to `content`/`color`/font properties/`direction` (all of which PeachPDF fully supports on `::marker`, including real per-item numbering via the `list-item` counter and `<ol start>`/`<ol reversed>`/`<li value>`) — no browser implements marker-box sizing either, for the same reason. See [Pseudo-elements](docs/html-css-support.md#pseudo-elements).
- Tagged PDF: anonymous (CSS-generated, e.g. `display: table-cell` on a `<div>`) table structure cannot have its `TR`/`TH`-or-`TD`/`THead`/`TBody`/`TFoot` tagging overridden via `-peachpdf-pdf-tag-type` — the synthesized anonymous boxes have no source element for any selector to match. Real `<table>`/`<tr>`/`<td>` markup is required for override control. See [Tagged PDF (PDF/UA) Support](docs/html-css-support.md#tagged-pdf-pdfua-support).
- Per-character font matching (`@font-face` `unicode-range` + glyph-coverage fallback across the `font-family` stack) is implemented (issue #158). Astral/supplementary-plane codepoints (above U+FFFF, i.e. emoji) are now supported via the cmap **format-12** subtable — glyph lookup (`OpenTypeDescriptor.CharCodeToGlyphIndexCore` → `CMap12.MapCodeToGlyph`) and coverage (`CMapCoverage.Extract(CMapTable)`) both consult it, so boundary (2) of the old #172 note is closed for monochrome-outline emoji. Remaining boundaries: (1) fallback walks only the declared `font-family` stack plus the default font, not a last-resort scan of every installed font, so a codepoint no declared family covers still shows a missing-glyph box (still [issue #172](https://github.com/jhaygood86/PeachPDF/issues/172)); (2) **color** emoji (`COLR`/`CPAL`, `CBDT`/`CBLC`, `sbix`, SVG-in-OpenType) is unsupported — the pipeline embeds `glyf`/`CFF` outlines only, so a color-emoji font renders blank/monochrome ([issue #175](https://github.com/jhaygood86/PeachPDF/issues/175)); (3) cmap format-14 variation sequences aren't applied, and a font with *only* a format-12 subtable (no format-4) still isn't usable (`CMapTable.Read` requires format-4 for the WinAnsi `Widths` loop). Overlap tie-break for two ranged faces covering the same codepoint is last-declared-wins (CSS cascade order); real subsets are disjoint so this is an edge case. See [Per-character font matching](docs/html-css-support.md#per-character-font-matching-and-coverage-fallback), `UnicodeRangeLayoutIntegrationTests`, `FontResolverCodepointTests`, `Format12CmapAstralTests`.
- No text shaping: PeachPDF maps each codepoint 1:1 to a glyph via `cmap` with no OpenType Layout (`GSUB`/`GPOS`), no Unicode Bidi Algorithm, and no Arabic/complex-script joining. So automatic ligatures (`f`+`i`→`ﬁ`), codepoint-less designer ligatures, `font-variant-ligatures`, real bidi reordering (only `direction:rtl` word-box mirroring via `CssLayoutEngine.ApplyRightToLeft` exists; `unicode-bidi` is inert), Arabic contextual forms, Indic reordering, and kerning are all unsupported. `GlyphSubstitutionTable.Read` is an empty stub; there is no `GPOS` parser. Filed as [issue #176](https://github.com/jhaygood86/PeachPDF/issues/176). Current behavior is pinned by `ShapingCharacterizationTests` (characterization, not desired output). See [Text shaping](docs/html-css-support.md#text-shaping).
- Paint order (`CssBox.PaintImpCore`'s stacking loop) correctly orders block → float → inline-level content per CSS2.1 Appendix E *within one stacking context's own direct participants* (including a plain block wrapper whose entire content is inline, e.g. a `<div>` around nothing but a resolved inline `<object>`/image — see `CssBox.ActsAsInline`). A float hoisted (`DomUtils.NeedsStackingHoist`/`FlattenStackingContext`) past its immediate container now preserves correct local order against its true, non-hoisted block/inline siblings whenever that immediate container is itself positioned (`position: absolute`/`relative`/`fixed`/`sticky`, regardless of whether it has an explicit `z-index` — see `DomUtils.IsLocalOrderingScope`/`SearchForHoistableDescendants`'s `claimFloatsHere` parameter) — this is what Acid2's own `.eyes`/`#eyes-a`/`#eyes-b`/`#eyes-c` needed, since `.eyes` is `position: absolute` with no `z-index`. A genuine stacking-context descendant (real `z-index`, `opacity<1`, or a transform) nested inside such a box still correctly keeps escaping all the way to the nearest true stacking context, not trapped locally — z-index competition only has meaning there. The one remaining, narrower gap: a float whose immediate container is a *plain, non-positioned* wrapper (no `position` at all) still doesn't preserve local order against non-hoisted siblings — it hoists to the nearest true stacking-context ancestor instead, since a plain wrapper has no Appendix-E-recognized atomic-ordering role to claim it locally. Fixing that remaining case would need the hoisting mechanism to track and re-interleave against a plain wrapper's own non-hoisted siblings too, which touches the same float/clip-ancestor hoisting logic a prior round already hardened against a double-paint regression (`StackingContextPaintRegressionTests.cs`) — not attempted here. See `Acid2FeatureVerificationTests.NormalFlowBlockFloatAndInline_PaintInAppendixEOrder`'s doc comment for the historical context, and `Acid2RegressionTests.Eyes_FloatChild_OrdersLocally_NotHoistedPastNonStackingContextAncestor` for the fixed `.eyes`-shaped case.
- Acid2's own checkerboard-interlock trick (`#eyes-b`'s `background: fixed url(...)` and `#eyes-a object object object`'s `background: url(...) fixed 1px 0` — two identical 2×2 diagonal-transparency PNG tiles, offset by 1px, meant to interlock into solid yellow) renders correctly in the *majority* of the overlap region as of the `BordersDrawHandler`/`RImage.Interpolate`/float-margin/border-double-count fixes below, but a narrow (~1em-wide) band at the shared left clip edge of the two layers still shows the raw, un-interlocked checkerboard instead of solid yellow. Investigated and ruled out: paint order (confirmed correct — `#eyes-b` paints first, the inline object second, matching Appendix E), the PNG's `tRNS`-chunk color-key transparency (confirmed correctly decoded into a PDF `/Mask` stencil with byte-for-bit-correct values, verified by decompressing the actual embedded `/Mask` and image streams and hand-checking every pixel against the source PNG), and image smoothing (fixed separately below — was real, but doesn't explain this narrower residual band, which persists identically in both MuPDF and PDFium). - What *was* fixed (a distinct, larger problem from the bullet above): before the margin-truncation fix described next, `#top`/`.picture`'s real content didn't just land on a different page than `.intro` — it landed ~350-450pt *into* that second page, nowhere near the fixed bars either, because `#top`'s `margin-top: 100em` paginated straight through as literal blank vertical space (the page-slot-skip fix above only skips *wholly* blank pages; it doesn't stop content from resuming arbitrarily far into a *partially* blank one). Fixed per [CSS Fragmentation Module Level 3 §5.2](https://www.w3.org/TR/css-break-3/#break-margins) ("When an unforced break occurs before or after a block-level box, any margins adjoining the break are truncated to zero") — the same rule real browsers' print mode and Prince XML apply, confirmed by diffing PeachPDF's Acid2 output against Prince's own published Acid2 PDF sample directly. `CssBox.PerformLayoutImp`'s Static/Relative positioning branch now detects when a collapsed margin would push a box across one or more page boundaries by itself (as opposed to real content straddling a boundary, which `BreakInside: Avoid`/orphans-widows still handle separately) and, if so, discards the whole margin and starts the box flush at the top of the very next page instead — matching the same "land at `MarginTop` within the next page" convention the existing forced-break logic already used (`CssBox.cs`'s `IsForcedBreakValue` branch), just for *unforced* breaks. Margins after a *forced* break (`page-break-before: always` etc.) are explicitly excluded and still preserved un-truncated, per the same spec section. Two narrower, deliberately-not-attempted gaps in this fix, found by post-change review and not exercised by Acid2 itself (`#top`'s previous sibling is `div.intro`, never null): (1) a box that's the *first* child of a container whose own border/padding blocks CSS2.1 margin-collapse-through has no `prevSibling`, so the truncation guard (`prevSibling is not null`) skips it entirely — an oversized `margin-top` on such a box still paginates through as blank space; (2) the fix lives solely in `PerformLayoutImp`'s Static/Relative branch — `CssLayoutEngineFlex`/`CssLayoutEngineTable`/`CssLayoutEngineColumns` position their own children independently and aren't covered, so an oversized unforced-break-adjoining margin inside a flex/table/multicol context isn't truncated either.

- Margin collapsing (`CssBox.FoldSelfCollapsingMargins`, added by the Acid2 mouth-gap fix) folds every in-flow descendant's margins of a self-collapsing box into one adjoining set without checking for clearance — CSS2.1 §8.3.1 excludes margins separated by clearance from the adjoining set. Hitting the difference requires a float plus a cleared empty child *inside* a collapse-through box; deemed too exotic to special-case.
- A table with only a repeating `<tfoot>` (no repeating `<thead>`) whose first body row would cross a page boundary is caught by neither `CssLayoutEngineTable.LayoutCells`'s whole-table pre-checks: the headerless one requires `!_shouldRepeatFooters`, and the header-aware one (added by the keep-with-next fix below) requires `_shouldRepeatHeaders`. Narrow and not attempted — a real-world footer-only (no header) repeating table is uncommon, and the fix would need its own third pre-check variant keyed off `_footerHeight` instead of `_headerHeight`. Filed as [issue #151](https://github.com/jhaygood86/PeachPDF/issues/151).
- The header-aware whole-table pre-check (`CssLayoutEngineTable.LayoutCells`, added by the keep-with-next fix below) decides from `EstimateRowHeight`, the same one-line-of-text heuristic the headerless pre-check already uses — blind to tall block content inside the first row's cells, same as the pre-existing gap the headerless pre-check's own post-check (real-layout-based, run after the table is fully laid out) closes. That post-check is itself gated on `!_shouldRepeatHeaders`, so a repeating-header table whose *first row estimate* under-shoots (declares "fits" when real content doesn't) has no equivalent real-height fallback. Not attempted: extending the post-check to the headered case would need it to also correctly reposition the header's already-created `CssProxyBox` (which captures a paint-time position snapshot at proxy-creation time — see `CssProxyBox.PerformLayoutImp`), not just `_tableBox.Location`, unlike the headerless case. - **CSS Nesting** (issue #233 — [CSS Nesting 1](https://www.w3.org/TR/css-nesting-1/)): nested style rules and the `&` nesting selector now parse. **Resolved at parse time into ordinary flat rules**, so the matcher/cascade need zero nesting awareness. Pieces:
  1. **Classification without a token buffer** (`StylesheetComposer.FillDeclarations`, the line-~745 branch). A new `TryCreateNestedRule`/`IsNestedRuleAhead` scans the upcoming construct bracket-depth-aware for the first top-level `{` (→ nested rule) vs `;`/`}`/EOF (→ declaration). This resolves the `div { }` (type selector, first token is an Ident like a property name) and `a:hover { }` (ambiguous with `a: hover`) cases via `{`-before-`;`, not a fragile first-token heuristic. A token pushback buffer would corrupt hex colors: the lexer tokenizes `#123456` differently in value vs non-value mode (`Lexer.IsInValue`), and a buffered value replayed into `CreateValue` desyncs.   3. **`&` resolution** (`CreateNestedStyleRule`/`ResolveNestedSelector`). The nested prelude's source text is sliced with the same raw marks and resolved against the enclosing rule's `SelectorText` (nearest `StyleRule` on `_nodes`): each `&` → `:is(<parent>)` (giving `&` the parent's specificity); a prelude with no `&` becomes `:is(<parent>) <prelude>`, which uniformly yields both the implicit-descendant (`.b` → `:is(p) .b`) and leading-combinator (`> .b` → `:is(p) > .b`) forms. The `&` substitution is **string-aware** (a char scanner tracking quote state), so a `&` inside an attribute-value string (`[data-x="a&b"]`) is preserved and doesn't wrongly flip the rule off the descendant branch — caught by post-change review. The resolved string is parsed via `Parser.ParseSelector` — `:is()` accepts complex/nested args (`MatchesFunctionState` feeds a full child `SelectorConstructor`), so deep nesting `:is(:is(.a) .b) .c` works. `SelectorConstructor` is untouched (resolved text has no `&`).
  4. **Storage + indexing.** `StyleRule` gained a `NestedRules` list (`IStyleRule.NestedRules`); the resolved nested rule is attached to its parent. `CssData.IndexRules` recurses into `NestedRules` in the **same** media/`@layer` context and `ref order`, so nested rules inherit the parent's context and sort right after it in document order. `CollectIndexKeys` already buckets the resolved selectors well (a `:is(...) .title` indexes under `.title`).
  See `CssNestingIntegrationTests` (implicit descendant, `&`-compound, type-selector & type:pseudo nesting, child combinator, multi-level, parent-list, hex-color regression, declaration-after-nested, `@media`/`@layer` inheritance, url-brace). **Accepted gaps** ([#253](https://github.com/jhaygood86/PeachPDF/issues/253)): nested at-rules (`@media`/`@layer`/`@supports` *inside* a style rule) aren't handled (place at top level); a custom-property value literally containing a `{` (e.g. `--x: { }`) is classified as a declaration but hits a pre-existing `CreateValue` brace limitation — neither occurs in real utility-framework output. `#233`-family efficiency note: a resolved subject hidden in `:is()` (e.g. `&:hover` → `:is(.card):hover`) buckets Universal (full-scan), correct but slower.

- **Keep-with-next stranding a heading before a relocated table/thead** (customer bug report): a section heading with `page-break-after: avoid` (the UA default for h1-h6 under `@media print`) could be left alone at the bottom of a page while the table immediately following it started on the next page, via two gaps neither covered by the pre-existing keep-with-next mechanisms (`DomUtils.GetPrecedingKeepWithNextRun`, `CssBox.OffsetTopWithKeepWithNextRun`, and the table engine's headerless pre-check/post-check). Gap 1: `CssBox.PerformLayoutImp`'s margin-crossing unforced-break branch (Static/Relative positioning, the same branch as the margin-truncation fix above) relocated a box to the next page without pulling the preceding avoid-chained run along — fixed by mirroring the same run-pull/relax-if-unsatisfiable logic used elsewhere. Gap 2: the table engine's whole-table pre-check was gated on `!_shouldRepeatHeaders`, so a table with a repeating `<thead>` whose header fit on the current page but whose first body row didn't was never relocated — fixed with a second pre-check that runs after the header's height is known, deliberately not gated on the whole table body fitting one page (a long table starts fresh on the next page and fragments normally from there). The two table-engine pre-checks share their run-pull logic via `CssLayoutEngineTable.PullKeepWithNextRun`. See `PageBreakTableKeepWithNextIntegrationTests.cs`, including a test for the two fixes composing (both firing in sequence for the same table) without double-counting the offset.

## Thread safety

A `PdfGenerator` instance is not thread-safe — never call it concurrently or reuse one instance across overlapping renders. Use a separate instance per thread/request/batch item; see [Thread safety](docs/usage-examples.md#thread-safety) in usage-examples.md.
